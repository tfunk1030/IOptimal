# GTP Setup Builder — Deep Codebase Audit (2026-04-10)

**Branch audited:** `codextwo`  
**Scope:** All subsystems — solver workflow, calibration, pipeline, analyzer, learner, legal search, candidate generation.

---

## Executive Summary

The codebase is architecturally sound and physics-principled. The 6-step solver workflow, compliance-physics models, strict calibration gate, and driver-anchor pattern are strong foundations. However, four categories of issues threaten output correctness and user trust:

1. **CRITICAL runtime bugs** that crash or produce wrong results on active code paths (grid-search `NameError`, shock velocity percentile distortion, thermal/LLTD recommendation dead paths).
2. **HIGH-severity calibration integrity gaps** (learner LLTD proxy feeds solver corrections; roll-gain "calibration" without fitting; `auto_calibrate` accepts R²-failing models into disk; perch formula inconsistency in legal search).
3. **HIGH-severity architecture violations** (Step 6 provisional solve before Steps 4/5 in `materialize_overrides`; absolute N/mm modifier floors that corrupt Porsche/Acura scale setups).
4. **Systemic data contract breaks** (symptom strings mismatched between `diagnose.py` and `recommend.py`; lap selection inconsistency across `extract`/`segment`/`driver_style`; `lltd_measured` semantic contamination across multiple modules).

## Section 1: Critical Bugs (CRASH or SILENT WRONG OUTPUT)

### BUG-C1 — `driver_profile` NameError in grid-search path
**File:** `pipeline/produce.py:1086`  
**Severity:** CRITICAL — runtime crash when `--search-mode grid` is combined with a loaded IBT session.

```python
# Line 1085-1086 in produce.py:
if measured is not None:
    objective.set_session_context(measured=measured, driver=driver_profile)
```

The variable in scope in this function at this point is named `driver` (not `driver_profile`). `driver_profile` is only defined in the `reason.py` orchestrator path. When the grid-search branch runs with telemetry available, Python raises `NameError: name 'driver_profile' is not defined`. This is a complete crash of the legal-search pipeline.

**Fix:** Change `driver=driver_profile` to `driver=driver`.

### BUG-C2 — Shock velocity percentile distribution distortion
**File:** `analyzer/extract.py:464-470`  
**Severity:** CRITICAL — corrupts all shock velocity statistics used by the damper solver and track profiling.

For heave+roll suspension architectures (Acura, Porsche in certain modes), the code synthesizes per-corner shock velocities from heave ± roll channels, then computes front/rear percentiles:

```python
front_sv = np.concatenate([lf_sv, rf_sv])  # length 2N
rear_sv = np.concatenate([lr_sv, rr_sv])   # length 2N
state.front_shock_vel_p95_mps = float(np.percentile(front_sv, 95))
```

`np.concatenate` is correct for **pooling** two equivalent distributions. For per-corner shocks from a heave+roll architecture, `lf_sv = |hf + froll|` and `rf_sv = |hf - froll|` are correlated (same heave base). Concatenating head-to-tail produces a bimodal distribution where the percentile calculation is **technically valid but physically misrepresents** the track bump profile because the two signals are time-synchronized and should instead be analyzed as max(lf, rf) or mean(|lf|, |rf|) to get a representative single-axis profile. The p95 of the concatenated version will be close to the maximum of the two p95s (slightly underestimating it), while the p50 is biased down. This affects track profile shock stats and damper click targets.

**Fix:** Use `np.maximum(lf_sv, rf_sv)` (element-wise max) and then take the percentile, or use the pooled array but document that this intentionally represents the combined demand. The current code gives a p95 that is neither the worst-corner p95 nor the average; it is an unintended arithmetic artifact.

### BUG-C3 — Thermal recommendation paths never trigger
**Files:** `analyzer/recommend.py:651`, `analyzer/diagnose.py:921-942`  
**Severity:** CRITICAL — thermal recommendation logic is completely dead in all sessions.

`diagnose.py` generates thermal problems with symptom strings of the form:
- `"LF spread +5.1C vs target +8C (inside too hot)"`
- `"RF spread -3.2C vs target +8C (too flat / outer loaded)"`

`recommend.py` checks for these with:
```python
if "inner hot" in problem.symptom.lower():   # NEVER matches — diagnose says "inside too hot"
elif "outer hot" in problem.symptom.lower():  # NEVER matches — diagnose says "too flat / outer loaded"
```

Neither string appears in any diagnosis output. The entire thermal recommendation chain (`camber_correction`, `toe_correction`, `pressure_correction`) is unreachable.

**Fix:** Change `recommend.py` checks to match the actual symptom strings:
```python
if "inside too hot" in problem.symptom.lower():
elif "too flat / outer loaded" in problem.symptom.lower():
```
Or alternatively standardize diagnose.py to use the shorter forms.

### BUG-C4 — LLTD recommendation path never triggers
**Files:** `analyzer/recommend.py:415`, `analyzer/diagnose.py:699-721`

`diagnose.py` generates LLTD problems with `category="balance"` and symptom strings containing `"Roll distribution proxy"`:

## Section 2: High-Severity Calibration Integrity Failures

### CAL-H1 — LLTD proxy contamination through learner pipeline
**Files:** `learner/empirical_models.py`, `learner/observation.py`, `solver/learned_corrections.py`  
**Severity:** HIGH — learner silently encodes a meaningless signal as LLTD correction.

`analyzer/extract.py` correctly documents that `lltd_measured = roll_distribution_proxy` — a geometric constant insensitive to spring stiffness. However:

1. `learner/observation.py` stores `lltd_measured` from the analyzer (the proxy value).
2. `learner/empirical_models.py:_fit_lltd_vs_arb` regresses this proxy against `rear_arb_blade`. Because the proxy is nearly constant (~0.50 ± 0.005), the "fit" has essentially zero slope, but the function still marks it as a `FittedRelationship` that can be queried.
3. `solver/learned_corrections.py` applies `lltd_measured_baseline` when `lltd_measured_mean > 0` (always true after 5+ sessions). This shifts the ARB solver's LLTD target using the proxy mean (~0.50), which contradicts the OptimumG physics target (0.521 for Porsche) set directly in `cars.py`.

The net effect: after 5+ Porsche/Algarve sessions, the learner overrides the carefully calibrated `cars.py` target with a geometrically-derived constant. The ARB driver-anchor then fires because model LLTD (0.391) doesn't match the now-wrong target (~0.50), and the "physics target unverifiable" justification looks wrong in the provenance trace.

**Fix:** In `learned_corrections.py`, add a guard: `if not getattr(empirical, "lltd_is_proxy", True):` before applying `lltd_measured_baseline`. Add `lltd_is_proxy: bool = True` to the empirical model's metadata and only set it `False` when true wheel-load telemetry is available (which per project docs it currently isn't).

### CAL-H2 — `auto_calibrate.py` writes models without R² floor enforcement
**File:** `car_model/auto_calibrate.py:620`  
**Severity:** HIGH — bad regression models can enter `models.json` and be loaded by the gate as "calibrated."

`_fit()` always sets `is_calibrated=True` regardless of the resulting R²:
```python
return FittedModel(
    ...
    r_squared=r2,
    is_calibrated=True,   # ← unconditional
)
```

The gate's 0.85 threshold is checked when **loading** the JSON, not when **writing** it. If `auto_calibrate` is run with sparse data (n < 8) or multicollinear features, it will write a low-R² model to disk. On the next pipeline run, the gate loads this model, sees `r_squared < R2_THRESHOLD_BLOCK`, and classifies the step as `weak` — but the user is not told that the model on disk is bad. It silently degrades output quality.

**Fix:** In `auto_calibrate.py`, after computing `r2`, check `if r2 < R2_THRESHOLD_BLOCK: is_calibrated = False` and write an appropriate log warning before returning. Alternatively, refuse to write to `models.json` if the regression doesn't meet the threshold.

### CAL-H3 — Roll gain "calibration" without fitting roll gains
**File:** `car_model/auto_calibrate.py:1273-1278`  
**Severity:** HIGH — geometry step can be marked calibrated with no new data.

When roll gradient CV < 30%, `auto_calibrate` sets `roll_gains_calibrated = True` and copies **existing** `cars.py` values for `front_roll_gain` / `rear_roll_gain` (with fallbacks 0.6 / 0.5). No regression is performed on these values from IBT data. The calibration gate then labels geometry as "IBT-calibrated" for Step 5, when the roll gains are simply hardcoded defaults, not measurements.

**Fix:** Remove the `roll_gains_calibrated` flag from this code path, or rename it `roll_gradient_stable` to accurately reflect what was validated. Only set `geometry_calibrated = True` for Step 5 when actual per-corner camber-loss measurements are available.

### CAL-H4 — `garage_model.py` uses raw spring index as N/mm for Ferrari/indexed cars
**File:** `car_model/garage_model.py:213-223`  
**Severity:** CRITICAL (for Ferrari/Acura) — m_eff corrections computed from wrong k_nmm.

```python
k_front_idx = setup.get("front_heave_index") or setup.get("front_heave_nmm")
...
k_nmm = float(k_front_idx)  # store raw index for car to decode
excursion_m = float(frh_std) * 2.33 / 1000.0
ratio_sq = (excursion_m / float(vp99f)) ** 2
self._update_physics_stat("m_eff_front_ratio", ratio_sq)
self._update_physics_stat("m_eff_front_input_k", k_nmm)
```

For Ferrari, `front_heave_index` is an integer (0–10 or similar). The code stores this index as `k_nmm` but labels it `m_eff_front_input_k`. Any downstream consumer that multiplies `k_nmm * ratio_sq` to get `m_eff_kg` will compute a wildly wrong mass (e.g., index=5 → k_nmm=5, m_eff = 5 × ratio ≈ 0.3 kg instead of 500 kg). Whether this actually corrupts solver corrections depends on whether the consumption path decodes before multiplying — the code comment says "store raw index for car to decode" but there is no evidence of that decode happening before use in `m_eff_front_ratio`. This needs tracing to confirm whether corrections reach the solver.

**Fix:** Either decode the spring rate immediately using the car's lookup table before computing `ratio_sq`, or refuse to store `m_eff` stats for indexed cars. Add an assertion `assert k_nmm > 10, "k_nmm looks like an index not a spring rate"` as a guard.

### CAL-H5 — Calibration gate docstring says "weak BLOCKS" but code does not block
**File:** `car_model/calibration_gate.py:35-37`  
**Severity:** HIGH (design/trust mismatch) — users reading the docstring will incorrectly assume weak-flagged steps do not produce output.

The `SubsystemCalibration` docstring states:
```
- "weak": ... Step BLOCKS by default.
```

But `check_step()` explicitly does NOT block on weak:
```python
if report.weak_block:
    report.blocked = False  # weak does not cascade
```

Steps with R² < 0.85 run and produce output, flagged only with `[~~]`. This is probably the right behavior (warn, don't stop), but the docstring creates a false expectation that has likely confused multiple implementers and will confuse future ones.

**Fix:** Update the docstring to say "Step still runs but outputs carry [~~] flag and WEAK CALIBRATION DETECTED banner. Weak blocks do NOT cascade to downstream steps."

## Section 3: High-Severity Workflow Architecture Issues

### ARCH-H1 — Step 6 (dampers) runs before Steps 4 and 5 in `materialize_overrides`
**File:** `solver/solve_chain.py:1244-1255`  
**Severity:** HIGH — violates the stated 6-step dependency chain.

Inside `materialize_overrides`, a provisional `DamperSolver.solve()` is called after rebuilding step2/step3, and the results are fed back as `front_hs_damper_nsm` / `rear_hs_damper_nsm` into the heave model. This creates a Step 6 → Step 2 coupling that:
- Runs dampers before ARB rates (Step 4) are resolved, meaning the damper zeta calculation uses spring rates but not the final ARB-adjusted k_roll
- Runs dampers before geometry (Step 5) which changes roll stiffness split
- Can cause oscillation if ARB or geometry changes significantly between the provisional and final damper solve

The main sequential path in `_run_sequential_solver` correctly orders Steps 1→6. The `materialize_overrides` provisional path reintroduces the dependency at a different point.

**Fix:** Either (a) remove the provisional damper call from `materialize_overrides` and accept that the heave model uses the previous iteration's HS damping estimate, or (b) explicitly document this as a "coupling approximation" with a comment explaining why it's acceptable (HS damping influence on heave excursion is second-order when springs are already stiff enough).

---

### ARCH-H2 — Modifier safety floors use absolute N/mm, break Porsche/Acura scale
**File:** `solver/modifiers.py:362-368`  
**Severity:** HIGH — Porsche heave spring ranges are 150–600 N/mm; the modifier floor of 40–60 N/mm is below the minimum garage value and may suppress the floor's intended effect without warning.

```python
if travel_pct >= 90.0:
    mods.front_heave_min_floor_nmm = max(mods.front_heave_min_floor_nmm, 60.0)
elif travel_pct >= 80.0:
    mods.front_heave_min_floor_nmm = max(mods.front_heave_min_floor_nmm, 50.0)
elif travel_pct >= 70.0:
    mods.front_heave_min_floor_nmm = max(mods.front_heave_min_floor_nmm, 40.0)
if pitch_range_deg > 1.5:
    mods.front_heave_min_floor_nmm = max(mods.front_heave_min_floor_nmm, 38.0)
```

For BMW (typical range 50–250 N/mm), a floor of 40–60 is meaningful. For Porsche (range 150–600), a floor of 60 N/mm is below the minimum legal value and does nothing. The heave solver will never produce 60 N/mm for Porsche; the floor is silently ineffective. More critically, the "safety re-apply" comment implies these floors were validated per-car — they weren't.

**Fix:** Make floors car-relative: `floor = max(car.heave_spring.front_min_nmm, car_fraction * car.heave_spring.front_min_nmm * 1.05)`. Or at minimum clamp the floor to `max(floor, car.heave_spring.front_min_nmm)`.

### ARCH-H3 — Perch computation formula inconsistency between legal search and garage model
**File:** `solver/legal_space.py:64,142`  
**Severity:** HIGH — perch offsets computed during legal search diverge from the garage model's perch model.

`legal_space.py` defines `FRONT_HEAVE_PERCH_K = 0.001614` (mm RH per N/mm heave from empirical fit) but then computes:
```python
front_heave_perch = front_perch_ref - delta_heave * 0.08
```

The coefficient `0.08` has **no documented origin** and is ~50× larger than `FRONT_HEAVE_PERCH_K` (0.001614). The constant was likely meant to scale from spring rate change to perch offset needed to maintain the same ride height — but the scaling factor should come from the inverse of the garage model's heave coefficient (~1/0.001614 ≈ 620, not 0.08). This means legal search candidates are assigned drastically wrong perch values relative to what the full solver computes, which corrupts the legality check for slider range, and forces the full solver to make large perch corrections after a candidate is selected.

**Fix:** Derive the perch sensitivity directly from `GarageOutputModel.front_coeff_heave_nmm` (or `front_coeff_inv_heave`): `delta_perch ≈ -delta_heave / front_coeff_heave_nmm` for linear models or the appropriate compliance equivalent.

### ARCH-H4 — Iterative DF balance correction overshoots
**File:** `solver/solve_chain.py:832-833`  
**Severity:** MEDIUM-HIGH — aero balance convergence can oscillate.

```python
corrected_target = inputs.target_balance + correction
# where correction = target - actual
# so corrected_target = inputs.target_balance + (target - actual) = 2*target - actual
```

This is a **full gain** correction (not a fractional step). If the first iteration produces `actual = 0.48` vs `target = 0.51`, correction = 0.03, corrected_target = 0.54. The rake solver at corrected_target = 0.54 may produce `actual = 0.55`, giving negative correction = -0.01, etc. With the 3-iteration limit it will usually converge, but for nonlinear aero maps (at vortex burst transitions) it can oscillate.

**Fix:** Apply a damping factor: `corrected_target = inputs.target_balance + 0.6 * correction`.

## Section 4: Medium-Severity Issues

### MED-1 — Lap selection inconsistency across pipeline modules
**Files:** `pipeline/produce.py`, `analyzer/driver_style.py`, `analyzer/segment.py`

The pipeline applies `min_lap_time` / `outlier_pct` filtering in `produce.py` to select laps for telemetry extraction. `analyzer/driver_style.py:195` calls `ibt.best_lap_indices()` without any filtering — it always uses the raw IBT best lap, which can be a different lap than the one used for shock velocity extraction, RH extraction, and corner segmentation.

This means the driver style profile (trail braking depth, throttle R², lap consistency) may describe a different lap than the setup metrics (shock p95, front RH std, understeer index). When modifiers use both together (e.g., smoothness scale on damping adjustments + heave target from RH analysis), they can combine mismatched data.

**Fix:** Pass the selected lap index through to `analyze_driver()` so all analysis uses the same lap.

### MED-2 — `detect_car_adapter` in `setup_registry.py` defaults unknown cars to BMW
**File:** `car_model/setup_registry.py:583-618`  
**Severity:** MEDIUM — if ever called for Porsche/Acura, wrong XML mapping applied.

```python
def _car_name(yaml_path: str) -> str:
    ...
    return "bmw"   # ← default fallback
```

This function is currently unused (no callers found in grep), but it sits as a trap for future callers. Any new code that calls `detect_car_adapter(yaml_path)` without checking the return value will silently apply BMW garage schema to any car whose YAML doesn't clearly contain BMW-specific keys.

**Fix:** Remove the function or raise `ValueError("Cannot auto-detect car type from YAML structure")` instead of returning `"bmw"`.

### MED-3 — Knowledge store track key collisions
**Files:** `learner/ingest.py`, `learner/recall.py`

All learner files use `track_name.lower().split()[0]` as the track key. "Sebring International Raceway" and "Sebring Motorcycle Complex" (hypothetical) would share `"sebring"`. More concretely, any track whose name starts with a common word (e.g., "Circuit de") would collide. The delta detector and empirical model files are named `bmw_sebring_delta_*.json` — a car competing at two tracks that share a first word would contaminate each other's empirical models.

**Fix:** Use a slugified full name (e.g., `"sebring-international-raceway"`) or the track's canonical `registry.id` field as the key.

### MED-4 — Delta detector uses ingest timestamp, not IBT session timestamp
**File:** `learner/observation.py`

`build_observation()` uses `datetime.now()` as the observation timestamp. Time-decay weighting (`TIME_DECAY_BASE ** days_since`) will weight recently-ingested old sessions as if they were new. If a user bulk-imports 6 months of old IBTs in one batch, every session gets today's timestamp and decays as if it's current data for the next 30 days. The solver will treat a 6-month-old setup insight as equally fresh as today's.

**Fix:** Extract the IBT session start time from the IBT header (it's available in the session_info YAML as `WeekendInfo.SessionInfo.Sessions[0].SessionNum` start time or from the file modification date as a minimum), and use that as the observation timestamp.

### MED-5 — `sensitivity.py` hardcodes BMW/Sebring targets for all cars
**File:** `solver/sensitivity.py`

DF balance reference `50.14`, heave sigma targets `8.0`/`10.0`, and input uncertainty table labeled "BMW Sebring" are hardcoded in `sensitivity.py`. If `build_sensitivity_report()` is called for Porsche/Algarve, it reports constraint proximity against wrong targets. For Algarve (different wing angle, different target balance), the "near binding" / "binding" classifications will be incorrect.

**Fix:** Accept `scenario_profile` and `solve_inputs` in `build_sensitivity_report()` and derive targets from them rather than hardcoding.

### MED-6 — `pushrod_geometry` always reported as calibrated
**File:** `car_model/calibration_gate.py:411-417`

A hardcoded comment says "All 5 cars now have calibrated pushrod models" and `status="calibrated"` is set unconditionally. No per-car check of whether the pushrod geometry model actually exists or has measured data. For Cadillac/Silverstone and Acura/Hockenheim (exploratory cars), this overstates Step 1 calibration coverage.

**Fix:** Add a `car.pushrod_calibrated` field (default `False`) and check it, or at minimum document that this claim needs per-car verification.

### MED-7 — ARB solver accepts `current_front_arb_*` but ignores them
**File:** `solver/arb_solver.py:241-244, 370-399`

`ARBSolver.solve()` accepts `current_front_arb_blade` and `current_front_arb_mode` as anchor parameters (parallel to the rear anchor that is used). The rear anchor fires and returns driver-loaded rear blade when model LLTD is far from target. The front anchor parameters are accepted but never consulted in the optimization loop. If the user's loaded front blade is meaningful, it's silently discarded while the rear is preserved.

**Fix:** Either implement the front anchor (symmetric with rear), or remove the parameters from the API and document that only rear anchoring is active.

---

### MED-8 — Bayesian optimizer uses synthetic proxy objective, not real physics
**File:** `solver/bayesian_optimizer.py`

The `_score()` function computes a hand-built heuristic score:
```python
lltd_est = weight_dist_front + 0.03*f_arb - 0.03*r_arb + 0.05  # arbitrary
```

This does not call the real `ObjectiveFunction` or the 6-step solver. The optimizer is finding the maximum of a surrogate that does not reflect actual vehicle physics — it can produce setups that look good under the proxy but are suboptimal under the real objective. The `BayesianResult.improvement_pct` is relative to the same proxy score, not to any physical metric.

**Fix:** Replace `_score()` with a call to a lightweight version of the real `ObjectiveFunction.evaluate()`, or at minimum document prominently that the improvement % is proxy-relative. Gate the Bayesian optimizer behind a calibration requirement (the objective needs real R² ≥ 0.85 models to be meaningful).

---

### MED-9 — `auto_calibrate.py` LOO RMSE = 0 for n < 5
**File:** `car_model/auto_calibrate.py:601-610`

When `n < 5`, LOO cross-validation is skipped and `loo_rmse` is set to `0.0`. A model fit to 4 data points with `loo_rmse = 0.0` looks artificially perfect in the provenance report. For Ferrari (12 observations, 9 for some steps) this may be fine, but for any new car starting calibration, the first few sessions will show loo_rmse = 0.0 which implies zero generalization error.

**Fix:** Set `loo_rmse = float("inf")` (or `nan`) when LOO is skipped. Surface this in the provenance output as "LOO RMSE: N/A (n < 5)".

---

### MED-10 — `setup_registry.py` default car = BMW for unspecified callers
**File:** `car_model/setup_registry.py:565,574`

`yaml_path_to_canonical()` and `sto_param_id_to_canonical()` default `car="bmw"`. If called without an explicit car argument (which is common for convenience wrappers), they silently apply BMW parameter mappings to any car. This is a systemic risk for any new code that forgets to pass `car=`.

**Fix:** Change defaults to `car: str | None = None` and raise `ValueError("car argument required")` when `None`.

## Section 5: Low-Severity Issues and Technical Debt

### LOW-1 — `R2_THRESHOLD_WARN = 0.95` defined but never used
**File:** `car_model/calibration_gate.py:60`

This constant was presumably intended to issue a warning band between 0.85 and 0.95 but `_classify_r_squared()` uses internal thresholds (0.90/0.70) that don't reference this constant. Dead code that implies a feature not implemented.

### LOW-2 — `t_avg_m` computed but unused in `wheel_geometry_solver.py`
**File:** `solver/wheel_geometry_solver.py:183-188`

`t_avg_m = (front_track_width_m + rear_track_width_m) / 2` is computed inside `_body_roll_at_g()` but never used in the function body. Likely a leftover from an earlier formula that was refactored.

### LOW-3 — `sigma_tol_mm` parameter in `solve_chain.py` unused in convergence check
**File:** `solver/solve_chain.py:752, 818-821`

The convergence loop in `_iterative_coupling_refinement` checks `df_tol` and `lltd_tol` but never `sigma_tol_mm`. The parameter exists in the signature but contributes nothing to convergence decisions.

### LOW-4 — `_num()` in `modifiers.py` swallows all input errors silently
**File:** `solver/modifiers.py`

`_num(x, default=0.0)` returns the default for any non-numeric input including `None`, empty strings, and exceptions. A missing telemetry field becomes `0.0`, which can be misinterpreted as "zero g lateral load" or "zero understeer angle." The function should at minimum log a debug warning when it falls back.

### LOW-5 — 500+ `getattr` calls remain in candidate_search.py, bmw_rotation_search.py, bmw_coverage.py
**Files:** `solver/candidate_search.py` (188), `solver/bmw_rotation_search.py` (113), `solver/bmw_coverage.py` (78)

Most are legitimate optional-feature checks (car-type branching), but the density of `getattr` in `candidate_search.py` means any new car-specific field that the search should use will silently fall back to the default rather than raising an attribute error. This makes extending the candidate search for a new car invisible without grepping for each getattr.

### LOW-6 — Knowledge store has no file locking
**File:** `learner/knowledge_store.py`

Documented in CLAUDE.md as a known limitation. The desktop app runs watcher + sync + pipeline concurrently. If a watcher event triggers `ingest_ibt` while `pipeline.produce` is writing an observation, both write to the same JSON concurrently and the faster writer's data is lost. This is not a hypothetical edge case — it happens every time a new IBT is completed during an active session.

**Fix:** Use `fcntl.flock()` on Unix or a `.lock` file pattern around all JSON writes in `knowledge_store.py`.

### LOW-7 — `cars.py` RideHeightModel default `is_calibrated = True`
**File:** `car_model/cars.py:261`

The `RideHeightModel` dataclass defaults `is_calibrated=True`. Any careless instantiation (e.g., `RideHeightModel()` with no arguments) produces a model that claims to be calibrated with all-zero coefficients. The front calibration check `front_is_calibrated` also only tests the linear heave/camber coefficients, not the compliance terms, so a compliance-only front model (BMW front fit form) could be misjudged as uncalibrated.

### LOW-8 — Porsche legal range inconsistency: legality.py vs garage_params.py
**Files:** `car_model/legality.py:227`, `car_model/garage_params.py`

`legality.py` sets `rear_heave_max: 1000.0` for Porsche while `garage_params.py` Porsche third spring max is 300 N/mm. These should agree. If the legality checker validates a candidate's rear third spring of, say, 400 N/mm against the 1000 ceiling, it passes legality but is an illegal garage value.

## Section 7: Calibration System Assessment

### What is working well

- **Strict gate architecture**: The `calibrated` / `weak` / `uncalibrated` + cascade rules are correct and enforced. The 6-step dependency chain (5→4, 6→3, 3→2, etc.) matches actual data flow.
- **Compliance physics**: The 1/k feature selection for static RH models took Porsche rear RH from R²=0.61 to R²=0.94 — the single biggest accuracy improvement in the system.
- **Provenance tracking**: Every run produces a full provenance block with R², source, and confidence level per subsystem. This is production-quality.
- **Porsche calibration**: 6/6 steps clean, driver-match 14 exact / 7 close on newest IBT. The σ-calibration + sticky pre-check + driver anchor chain is architecturally elegant and physics-principled.
- **BMW calibration**: 6/6 steps, aero compression V²-RMS correction, compliance physics, auto-calibrate producing meaningful coefficients.

### What needs improvement

- **LLTD calibration gap**: No true LLTD measurement exists. The current OptimumG formula target (0.521) is physics-principled but cannot be ground-truthed. The learner actively undermines it by feeding the proxy back as a "measured" baseline. Priority: implement `lltd_is_proxy` guard immediately.
- **Ferrari/Cadillac/Acura calibration**: Ferrari is at 3/6 (corner spring, LLTD, ARB partially blocked). Cadillac and Acura are exploratory. The system correctly blocks and instructs for these — the architecture works but needs sessions.
- **Roll gain measurement**: No car currently has roll gains fitted from IBT. All cars use hand-calibrated or defaulted values. `auto_calibrate` marks these as "IBT-calibrated" which is misleading.
- **ARB calibration quality**: All cars rely on hand-calibration for ARB. Auto-cal can back-solve total roll stiffness but cannot split front/rear independently. The Porsche ARB sits at `MEDIUM` confidence (signal-below-noise).

## Section 8: Workflow Accuracy Assessment

### Step 1 (Rake / Ride Heights)

**Accuracy: MEDIUM-HIGH for BMW/Porsche, LOW for others.**

The rake solver correctly applies compliance physics, V²-RMS speed reference (not median), and reconciles static RH after heave/corner spring selection. The `solution_from_explicit_offsets` fix (April 7) closed the static drift bug. The remaining gap is the 5 mm rear pushrod error on newest Porsche IBT (1.3 mm in rear static RH propagates to 5 mm in pushrod). The rear RH dynamic anchor would close this.

### Step 2 (Heave / Third Springs)

**Accuracy: HIGH for BMW/Porsche.**

σ-calibration with sticky pre-check, cal_ratio correction, and driver-anchor chain produces exact matches on BMW and Porsche driver setups. The Gaussian σ assumption is a known simplification. The `min_rate_for_sigma` function architecture is the strongest physics implementation in the codebase.

### Step 3 (Corner Springs)

**Accuracy: MEDIUM for BMW/Porsche.**

The rear coil spring driver anchor (copy driver-loaded rate) is too aggressive — it prevents the solver from recommending a different spring when the physics model would suggest one. The Porsche csm mutation (side effect on car model during solve) is a re-entrancy risk for parallel candidate evaluation.

### Step 4 (ARBs)

**Accuracy: LOW for all cars (no true LLTD measurement).**

The OptimumG formula target is physics-principled but unverifiable. The driver anchor fires for Porsche (model says 0.391, target 0.521, 13 pp gap) and produces the correct garage blade — but only because the driver's blade is loaded, not because the physics agrees. This is the biggest accuracy gap in the system.

### Step 5 (Geometry)

**Accuracy: MEDIUM.**

Roll gain values are hardcoded defaults. Camber optimization over the roll range is theoretically sound but uses split-axle approximation. No test compares predicted vs measured tire temps at different camber settings.

### Step 6 (Dampers)

**Accuracy: MEDIUM-HIGH for Porsche (88 click-sweep sessions), MEDIUM for BMW.**

The zeta calibration from real click-sweep data is the gold standard approach and gives high accuracy for Porsche. BMW medium confidence. The provisional Step 6 before Steps 4/5 in `materialize_overrides` introduces a small but real physics inconsistency.
